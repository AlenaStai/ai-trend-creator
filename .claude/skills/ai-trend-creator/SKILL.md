# AI Trend Creator — работа с репозиторием

Личный инструмент автора AI-блога для цепочки **Trends → Analyze → Generate Content →
Generate Video → Export**. Полная архитектура, границы MVP, стиль контента и план
разработки описаны в `CLAUDE.md` в корне репозитория — **прочитай его первым делом**,
если ещё не в контексте. Этот файл ниже не дублирует то, что там уже написано,
а описывает конкретные рабочие процессы.

## Структура проекта

```
app.py                    # точка входа Streamlit, 4 вкладки цепочки
collectors/                # сбор трендов: trends_mcp_client (основной), reddit_collector,
                            # youtube_collector (фолбэки), rss_collector
analysis/                  # trend_analyzer (скоринг), trend_explainer (веб-поиск сути тренда)
generation/                 # content_generator (сценарий), video_generator (HeyGen Video Agent)
export/exporter.py          # сборка финального пакета (видео + тексты)
utils/                      # db.py (SQLite), translator.py (перевод заголовков), speech.py (голос)
data/trends.db               # SQLite, не в git (.gitignore)
data/videos/                  # скачанные рендеры, не в git
export_output/                # готовые экспортированные пакеты, не в git
scripts/start_app.command      # быстрый запуск сервера + браузер
```

Каждый collector возвращает список словарей одинаковой структуры
(`title, source, url, score_raw, published_at`). Ошибка одного источника не должна
ронять остальные — везде try/except с фолбэком на следующий источник.

## Запуск сервиса

```bash
source .venv/bin/activate   # если venv ещё не создан: python3 -m venv .venv && pip install -r requirements.txt
streamlit run app.py
```

Открывается на `http://localhost:8501`. Для быстрого запуска без ручных команд —
`scripts/start_app.command` (или ярлык на рабочем столе).

Перед первым запуском нужен заполненный `.env` (скопировать из `.env.example`,
ключи и их источники расписаны в README.md, раздел «Переменные окружения»).
`.env` никогда не коммитить — он уже в `.gitignore`.

## Внесение изменений

- Сначала свериться с разделом «Границы MVP» в `CLAUDE.md` — не добавлять то, что
  явно вынесено за скоуп (автопостинг, fal.ai/InVideo, аналитика по опубликованному
  контенту), если пользователь явно не попросил выйти за MVP.
- Простой читаемый Python без лишней абстракции — это MVP, не продакшн-система.
- Если меняется что-то в `generation/content_generator.py` (промпт сценария) —
  результат должен проходить проверку по принципу `/humanizer` из `CLAUDE.md`
  (раздел «Стиль контента»): короткие рубленые фразы, самоирония, без симметричных
  конструкций и инфоблогерских CTA, хук в первые 2 секунды, явные паузы в разметке.
- Если меняется `generation/video_generator.py` — помнить, что HeyGen Video Agent
  по умолчанию трактует `--prompt` творчески; сценарий должен оборачиваться явной
  инструкцией использовать текст дословно (см. `CLAUDE.md`, раздел «MCP-серверы»).
- После значимых изменений в цепочке — проверить, не разошлось ли описание в README
  с реальным поведением (см. ниже).

## Проверка на ошибки

В проекте нет настроенного линтера/тестов (requirements.txt содержит только рантайм-
зависимости) — проверка держится на компиляции и реальном запуске:

```bash
# синтаксис всех модулей проекта
python3 -m py_compile app.py collectors/*.py analysis/*.py generation/*.py export/*.py utils/*.py

# импортируются ли модули без падения (ловит опечатки в именах/сигнатурах)
python3 -c "import app"
```

Дальше — по возможности реально погонять сервис (`streamlit run app.py`) по цепочке
Trends → Analyze → Generate Content, а не только проверить, что он стартует. Video-шаг
(HeyGen) долгий (20-45 минут) и тратит кредиты — гонять вживую только если явно нужно
проверить именно этот шаг, не по умолчанию при каждой правке.

## Подготовка к публикации на GitHub

Перед любым `git push` в публичный репозиторий — обязательный чек-лист секретов:

1. `.env` не должен быть в `git ls-files` и должен попадать под `git check-ignore -v .env`
2. `.gitignore` должен содержать как минимум: `.env`, `.venv/`, `data/*.db`,
   `data/videos/` (если добавлена), `export_output/`, `*.mp4`, `.DS_Store`
3. Проверить код на хардкод ключей (не только текущий диф, а весь `git ls-files`):
   ```bash
   grep -rnE "(api[_-]?key|secret|token|password)\s*=\s*['\"][A-Za-z0-9]" --include="*.py" .
   ```
   Все находки должны быть `os.getenv(...)`, не литералами.
4. Проверить историю на случай, если секрет коммитился раньше и позже был удалён:
   ```bash
   git log --all --diff-filter=A --name-only --pretty=format: | sort -u | grep -iE "\.env$|secret|credential"
   git log --all -p | grep -nEI "sk-ant-|AIza[A-Za-z0-9_-]{20,}|tmcp_live_"
   git fsck --full --unreachable --dangling   # висячие blob'ы от git add + reset без коммита
   ```
   Если что-то найдено — публикацию **останавливать** и предупреждать пользователя,
   секрет из истории так просто не убрать (нужен `git filter-repo`/BFG + ротация ключа).
5. Только после чистого результата всех проверок — коммит, затем публикация
   (`gh repo create ... --push` или `git push` в уже существующий remote).
   Создание публичного репозитория и push — необратимые публичные действия,
   не выполнять их без явного запроса пользователя на публикацию именно сейчас.
